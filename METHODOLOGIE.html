1. Portée du document et conventions

1.1 Ce que décrit ce document

  • Le modèle scientifique implémenté (cohérent avec SCIENCE.md).
  • La chaîne de calcul saisie UI → verdict GO/NOGO Monte-Carlo.
  • Les hypothèses simplificatrices, leurs domaines de validité.
  • Les évolutions : chaque limite identifiée → proposition chiffrée adossée à la recherche aéro ou à la DT FFP 49.

1.2 Ce que le document ne décrit pas

  • L'UI (DOM, Tailwind, Leaflet) — voir commentaires en tête de app.js, map.js, charts.js, ui-optimizer.js.
  • La réglementation pilote avion — voir METHODES_LARGAGE.md.

1.3 Conventions

  • Langue : français. Vocabulaire technique suit AERODYNAMICS_RESEARCH.md §7.
  • Unités : SI strict interne ; kt, NM, ft, FL, ° en UI. Conversions PhysicsCore.KT2MS, NM2M, DEG2RAD.
  • Repère : Est/Nord en m relatifs au centre DZ.
  • État : s = [xE, xN, z, vE, vN, vz]. z absolu (AMSL).

2.3 Duplication de physique — point critique

La physique existe en double. physics-core.js est canonique, utilisé par OptimizerV2, le worker V2 et MonteCarlo. simulation.js contient un jumeau legacy (getAtmosphere, rk4Step, windVecAtZ, runCoreSimulation…) utilisé par le chemin main-thread pour le rafraîchissement live.

Règle. Toute modification doit être appliquée aux deux fichiers simultanément, et SCIENCE.md tenu à jour. Le §15.1 propose la suppression du jumeau — priorité n°1 des évolutions.

2.4 Un seul optimiseur

Le pipeline d'optimisation est fourni uniquement par OptimizerV2 (optimizer-v2.js), qui délègue le travail parallèle à optimizer-worker-v2.js. Le bouton "OPTIMISER" de index.html (onMainOptimize) appelle directement OptimizerV2.run().

2.5 Contrat Web Worker

Les workers reçoivent des entrées sérialisées (windProfile, jumpersList, nfzSerialized, cfg). Aucun DOM, aucun état partagé. Toute nouvelle entrée doit : (1) être ajoutée au message, (2) être prise en compte dans PhysicsCore, (3) être remontée côté ui-optimizer.js / optimizer-v2.js.

3. Modèle atmosphérique

3.1 ISA à 3 couches

CoucheAltitudeT(z)P(z)
Troposphère0–11 000 mT = 288.15 − 0.0065·z101325·(T/288.15)5.2559
Tropopause11 000–20 000 m216.65P₁₁·exp(−g·(z−11000)/(R·216.65))
Stratosphère> 20 000 m216.65 + 0.001·(z−20000)P₂₀·(T/216.65)−g/(0.001·R)

Constantes : g = 9.80665 m/s², R_air = 287.058, ρ₀ = 1.225 kg/m³, clamp T ≥ 150 K. isaDelta (K) décale uniquement T. ρ = P/(R_air·T).

3.2 Rôle actuel des données méteo pour ρ

Aujourd'hui, ρ(z) est calculée uniquement à partir de l'ISA + isaDelta UI, pas des données Open-Meteo. La chaîne est : 
UI                    PhysicsCore                      Sortie
────────────────────────────────────────────────────────────
altM (FL × 30.48)  ─┐
qnh (depuis méteo) ─┼─→ altQnhM = altM + (qnh−1013.25)·8.43
isaDelta (UI)      ─┼─→ getDensity(altQnhM, isaDelta) ─→ ρ(z)
                    │
windProfile[i].temp ╳  (stockée mais pas utilisée pour ρ)
windProfile[i].geoH ╳  (stockée mais pas utilisée pour ρ)
qnh est bien injecté dans l'altitude effective. isaDelta est à saisir manuellement (défaut 0). Les températures Open-Meteo par niveau sont stockées mais inutilisées côté physique — elles ne servent qu'à l'airgram.

Conséquence : en journée chaude (ISA+15 °C en Occitanie) sans ajustement UI, ρ est faussée de ~5 %, décalant TAS et vT d'autant. §15.13 propose la dérivation complète de ρ(z) depuis le profil thermique Open-Meteo.

3.3 LUT densité

Float64Array(1301) précalculée à l'initialisation : 0–13 000 m pas de 10 m, interpolation linéaire. Construite à isa = 0 ⇒ isaDelta ≠ 0 court-circuite la LUT. Pour les optimisations massives, garder isaDelta = 0.

3.4 Correction QNH

altQnhM = altM + (qnh − 1013.25) × 8.43

8.43 m/hPa = gradient altimétrique standard au sol (pH ≈ R·T/g ≈ 8 430 m).

3.5 TAS

TAS = KIAS × √(ρ₀ / ρ(z))

FL140 ISA std : ~+20 %. KIAS 120 → TAS ~144 kt.

4.4 Interpolation vectorielle du vent — justification physique

Dans PhysicsCore.windAtZ(z), l'interpolation porte sur les composantes E/N précalculées, pas sur (vitesse, direction).

Pourquoi vectoriel ? Interpoler la direction à travers 360°/0° donne des résultats absurdes. Exemple : 355° et 5° interpolés → 180° (à l'envers) alors que l'interp E/N donne 0° (correct).

Pourquoi linéaire et pas spline ? Le vent en troposphère libre varie de façon douce avec l'altitude, et la résolution verticale Open-Meteo (~450 m entre couches en basse tropo) donne une erreur d'interp linéaire de l'ordre de grandeur de l'incertitude modèle (1–2 kt, quelques degrés). Une spline cubique n'apporterait rien physiquement.

Physiquement valide ? Oui pour la troposphère libre (vent quasi-géostrophique, variation lente). Dans la couche limite de surface (< 300 m), c'est le profil log de Prandtl qui prend le relais — c'est lui qui est physiquement important, pas la spline entre couches d'altitude.

Cohérence avec les données source. Open-Meteo fournit des moyennes par niveau pression, pas des profils haute résolution type radiosondage. La linéarité entre couches est donc le niveau de granularité approprié. Le cache eager §4.6 gèle les valeurs à l'init pour supprimer le coût à la requête.

4.5 Profil logarithmique de surface (Prandtl)

V(z) = V_ref × ln(z/z₀) / ln(z_ref/z₀)

Sous la plus basse couche ; z₀ = 0.03 m (terrain plat/herbe). Sous z₀ : vitesse nulle. Exemple : 100 m sol avec 15 kt à 300 m → ~12.7 kt.

4.6 LUT de vent précalculée (eager, comme ρ)

Exactement comme la LUT densité (§3.3), le profil de vent est transformé en deux Float64Array remplies eager à chaque setWindProfile : _wLutE[0..maxAlt] et _wLutN[0..maxAlt], par pas de 1 m (typiquement 0–14 000 m).

function windAtZ(z) {
  var k = z | 0;
  if (k < 0) k = 0;
  if (k >= _wLutE.length) k = _wLutE.length - 1;
  var f = z - k;
  return {
    e: _wLutE[k] + f * (_wLutE[k+1] - _wLutE[k]),
    n: _wLutN[k] + f * (_wLutN[k+1] - _wLutN[k])
  };
}

Coût mémoire. 2 × 14 000 × 8 B = 224 kB — négligeable.

Gain. windAtZ devient une lecture pure en O(1), sans recherche binaire, sans branchement log-Prandtl (déjà encodé). Retire une fraction significative du temps d'un simPass qui appelle windAtZ plusieurs milliers de fois par parachutiste.

Invalidation. Reconstruction à chaque setWindProfile (DZ, modèle, heure, perturbation MC). 14 000 évaluations ≈ 1 ms.

État actuel. Le cache est aujourd'hui lazy (sentinelle -999999, remplissage à la première requête). L'évolution §15.14 passe en eager pour le gain perf et la simplicité.

5. Hypothèses de largage et repères géométriques

5.1 Chaîne des repères

  1. Cible TGT (currentTarget en E/N).
  2. Offset cross-track : tgtE = targetE + crossM·perpE ; tgtN = targetN + crossM·perpN.
  3. Top vert : à topNm devant tgt.
  4. p1Exit : à delaiTopVert s après top vert.
  5. Sortie i : p_i = p1Exit + trackE·(i·dtSortie·gsMs), …
  6. Top rouge : dernière sortie.

5.2 Vitesse sol sur jump run

tw = wJump · track → gsMs = max(1, tasMs + tw) → dtSortie = espacementM / gsMs

5.3 Direction break-off et séparation intra-groupe

Signe du cross-track ⇒ côté RV ⇒ direction break-off. Groupes nbPara > 1 : sous-parachutistes distribués sur un cercle de rayon sepDist.

5.4 Hypothèses implicites

  • Axe constant, pas de correction dynamique.
  • TAS fixée, pas de pilotage avion modélisé.
  • Exit vertical parfait avec vitesse initiale = vitesse sol avion. Le §15.8 ajoute une incertitude sur le temps de sortie qui englobe les écarts.

6. Chute libre

6.1 Équations de traînée

Traînée paramétrée par vT (saisie vc) sans Cd·A explicite.

vr = v − w(z) − vtrack
c = (ρ(z)/ρ₀) × g / vT²
v̇E = −c·|vr|·vr,E, v̇N = −c·|vr|·vr,N, v̇z = −g − c·|vr|·vr,z

6.2 Limites du modèle

  • Traînée isotrope (c unique 3 directions) alors qu'un ventral a Cd·A vertical ~3× horizontal.
  • Portance wingsuit modélisée comme "vent fantôme" via trackVec.
  • vT unique pour toutes positions.

6.3 Injection du tracking

totalFallZ = env.altM − hOuvAbs
vtTrack    = vc × 1.15
grTrack    = min(trackDist / max(totalFallZ, 100), 1.5)    // ← borné à 1.5, sous-estime wingsuit
vhTrack    = grTrack × vtTrack
trackVec   = { e: sin(axe + trackAxis)·vhTrack, n: cos(axe + trackAxis)·vhTrack }
Le plafond grTrack ≤ 1.5 empêche toute wingsuit d'atteindre sa finesse réelle (2.5–3.5). Évolution §15.3.

6.4 Break-off

Si hBreak > hOuv, 2e phase de chute avec trackVec issu du vecteur radial _sepE/_sepN.

6.5 Intégration RK4

Pas dt = 0.25 s (0.15 s en validation finale V2). Erreur locale O(dt⁵), globale O(dt⁴).

6.6 Détection de seuil

frac = (s[2] − targetZ) / (s[2] − ns[2]) → s_exact = s + frac·(ns − s)

8. Vol sous voile

8.1 Modèle cinématique

vh,TAS = vzVoile × glide × √(ρ₀/ρ(z))
vz = vzVoile × √(ρ₀/ρ(z)) + kvirage·ψ̇²
vE = sin(ψ)·vh,TAS + wE(z), vN = cos(ψ)·vh,TAS + wN(z)

8.2 Correction densité

FL140 ≈ +20 % sur vz et vh.

8.3 Modèle de virage (dt = 0.5 s)

  1. Cible de cap targetH de steerFn.
  2. delta ramené [−180°, 180°].
  3. desiredRate = clamp(delta/dt, ±maxTR).
  4. Accélération angulaire ≤ rateAccel·dt.
  5. Freinage anticipatif si |delta| < stoppingAngle·1.2 && < 10° → taux × 0.7.
  6. Mise à jour du cap.
  7. vz += turnSinkFactor · currentRate².

Défauts : maxTurnRate = 30°/s, rateAccel = 90°/s², turnSinkFactor = 0.015.

8.4 Hypothèses implicites

  • Finesse constante — pas de freins/rear-riser (évol. §15.5).
  • Pas de décrochage.
  • Arrondi implicite à z = elevM.

9. Pilotage et steering

9.1 Steering vers un point (crab angle)

  1. dh = √(dx² + dy²). Si < 5 m cap figé.
  2. Vent w = windAtZ(z).
  3. wp = w · perp(dx̂, dŷ).
  4. Si |wp| ≥ vhMs → vol face au vent pur.
  5. Sinon comp = √(vh² − wp²).

Bruit pilote : noiseSeed · 20°·(1−skill). Délai réaction : 2.0/max(skill,0.1) s.

9.2 Steering deux-phases

t < tPilotCommon (8 s) : cap perpendiculaire (break-off). Après : direct RV.

9.3 Critère RV

Rectangle rvL × rvW (200×100 m), plancher rvAlt = 300 m AGL.

11.5 Ellipse de confiance 95 %

  1. Moyenne (mx, my).
  2. Covariance 2×2 Σ.
  3. Valeurs propres analytiques λ₁,₂ = tr/2 ± √(tr²/4 − det) (plancher 0.01).
  4. Angle principal ½·atan2(2σEN, σEE − σNN).
  5. χ² = −2·ln(0.05) ≈ 5.991 (2 ddl).
  6. a = √(χ²·λ₁), b = √(χ²·λ₂).
  7. Polygone 42 points, rendu Leaflet.

12. Optimiseur en 4 phases

12.1 Phase 1 — Scan grossier (parallèle)

  • Axes : ±60°/5° autour du vent + 0–360°/15°.
  • Offsets : −1.0 à +1.0 NM / 0.2.
  • Workers : 2–8.

12.2 Phase 2 — Raffinement

Top 8 au nPara max → axe ±5°/1° × offset ±0.15 NM/0.05.

12.3 Phase 3 — Ordre de sortie (AG)

Ordre initial — nouvelle règle (évolution §15.9). L'ordre classique supprime le hop&pop (non utilisé dans la plupart des sticks réels) et répartit les trackers à trois positions du stick (début, milieu, fin) pour aérer le trafic sous voile :

belly_big → [tracking_1] → belly_small → freefly_big → freefly_small →
aff → [tracking_2] → tandem → [tracking_3 / wingsuit]

Règle de répartition :

  • 1 tracker → placé en fin de stick.
  • 2 trackers → 1 au milieu, 1 en fin.
  • ≥ 3 trackers → 1 en début, 1 au milieu, le reste en fin.
  • Wingsuits toujours en dernier (DT FFP 49 : vitesse verticale faible + finesse élevée + trajectoire réservée).

Paramètres AG

  • Population popSize = min(20, max(8, 2N)).
  • Générations : 100 (N ≤ 7) / 300 (N > 7).
  • Sélection élitiste 50 %.
  • Mutation : swap aléatoire.
  • Pas de crossover (évol. §15.15 : Order Crossover OX).

12.4 Phase 4 — Validation Monte-Carlo

Config optimale + ordre soumis à MC avec paramètres configurables. Validation finale via simPass avec dtCfg.ff = 0.15 s.

15. Limites actuelles et plan d'amélioration

15.1 Unifier la physique (supprimer simulation.js)

function _selectRegime(stateE, stateN, tgtE, tgtN, w, vh_full) {
  var dx = tgtE - stateE, dy = tgtN - stateN;
  var dh = Math.sqrt(dx*dx + dy*dy);
  if (dh < 1) return 'full_flight';
  var wp = w.e * (dx/dh) + w.n * (dy/dh);   // + = tailwind
  if (wp >  0.4*vh_full)                      return 'min_sink';
  if (wp < -0.5*vh_full && wp > -vh_full)     return 'rear_riser';
  return 'full_flight';
}

15.8 Monte-Carlo : refonte des sources d'incertitude

Remplace la version initiale (altitude avion / ouverture dure / tour de suspentes écartés). Les incertitudes ci-dessous dominent réellement la dispersion observée en largage sport.

(a) Temps entre les sorties — climb-out + taille de groupe

Actuellement dtSortie déterministe. En réalité :

  • Solo propre : valeur nominale ± σ ≈ 0.5 s.
  • Groupe 2–4 : climb-out 3–5 s additionnel (mise en place sur la barre avant countdown).
  • Groupe 5+ : climb-out 5–8 s, dispersion large.
function perturbExitTimes(jumpers, dtNominal) {
  return jumpers.map(function (j) {
    var nb = j.nbPara || 1;
    var climbOutMean =
      nb === 1 ? 0 :
      nb <= 4 ? 2.0 + 0.5 * nb :   // 2.5 s @ 2, 4 s @ 4
                5.0 + 0.6 * nb;    // 8 s @ 5, 11 s @ 10
    var climbOutSigma = 0.4 + 0.15 * nb;
    var jitter   = gaussRandom() * 0.6;   // σ = 0.6 s (6 s annoncé → 5.4–6.6 s)
    var climbOut = climbOutMean + gaussRandom() * climbOutSigma;
    return { _climbOut: Math.max(0, climbOut), _exitJitter: jitter };
  });
}

Dans simPass : t_exit_j = delaiTopVert + Σ_{k<j} (dtSortie + climbOut_k + jitter_k) + jitter_j.

(b) σ vitesse terminale = 10 % (au lieu de 5 %)

Variabilité intrinsèque jour-à-jour (±5 %) + incertitude sur la catégorie typologique (±5 % de fourchette) ⇒ somme quadratique ~10 %.

(c) σ cap tracking ±20° (±10° wingsuit)

15.13 Dériver ρ(z) des températures Open-Meteo

Limite. ρ calculée ISA + isaDelta, les temp par niveau sont stockées mais inutilisées. Biais systématique 3–5 % ρ en atmosphère non-standard.

Proposition. LUT densité dérivée de la méteo à chaque fetchMeteo :

function buildDensityLUTFromMeteo(windProfile, maxAlt) {
  var sorted = windProfile.slice().sort((a,b) => a.z - b.z);
  var lut = new Float64Array(maxAlt + 1);
  for (var z = 0; z <= maxAlt; z++) {
    if (z <= sorted[0].z || z >= sorted[sorted.length-1].z) {
      lut[z] = getAtmosphereRho(z, 0);   // repli ISA hors plage
    } else {
      var lo = 0, hi = sorted.length - 1;
      while (hi - lo > 1) { var m = (lo+hi)>>1; if (sorted[m].z <= z) lo=m; else hi=m; }
      var f = (z - sorted[lo].z) / (sorted[hi].z - sorted[lo].z);
      var T = (sorted[lo].temp + 273.15) + f * (sorted[hi].temp - sorted[lo].temp);
      var p = (sorted[lo].hpa + f * (sorted[hi].hpa - sorted[lo].hpa)) * 100;
      lut[z] = p / (287.058 * T);
    }
  }
  return lut;
}

Cette LUT remplace la LUT ISA quand meteoData.time est l'heure courante. isaDelta UI devient accessoire (mode simulation manuelle sans méteo).

(c) Rendu — tubes d'incertitude

Plutôt que dessiner une ellipse à chaque bin (illisible), construire un polygone tube : enveloppe gauche-droite des ellipses successives, projetée sur la trajectoire moyenne :

function buildEnvelope(trajMean, ellipses, sigmaLevel) {
  var leftSide = [], rightSide = [];
  ellipses.forEach(function (ell, k) {
    var rad = sigmaLevel * Math.max(ell.a, ell.b);
    var heading = trajHeadingAt(trajMean, k);
    var perpE = -Math.cos(heading), perpN = Math.sin(heading);
    leftSide.push({ e: ell.cx + perpE * rad, n: ell.cy + perpN * rad });
    rightSide.unshift({ e: ell.cx - perpE * rad, n: ell.cy - perpN * rad });
  });
  return leftSide.concat(rightSide);  // polygone fermé
}

Deux tubes par parachutiste : 1σ rempli, 3σ contour seul ou rempli très transparent.

Combinable. Par défaut : ouverture/atterrissage ON, trajectoires OFF (lourd au-delà de 8 paras).

Coûts. Mémoire : N=100 × N_paras=15 × N_bins=30 × 2 × 8 B ≈ 720 kB. Rendu : ~1 800 points Leaflet — imperceptible.

Effet pédagogique. Permet de voir où la trajectoire est incertaine. Détecte les cas où "la marge moyenne est OK mais le 3σ touche une NFZ" — situation NOGO masquée par le percentile.

Fichiers. monte-carlo.js::run (échantillonnage temporel + ellipses par bin), monte-carlo.js::drawEnvelopes (nouvelle fonction), ui-optimizer.js (toggles).

15.19 Stick type pour l'optimisation

Limite. Le bouton "OPTIMISER" travaille sur le stick courant tel qu'il est dans jumpersList. Or :
  • À l'init, buildJumpers peuple jumpersList avec 4 ventral standard (vr_mv) si vide. C'est ce qui sera optimisé tant que l'utilisateur n'a pas explicitement composé son stick.
  • Pour explorer la configuration d'une DZ (axe optimal "type", vent type d'une saison), composer manuellement un stick représentatif est fastidieux.
  • Un stick type documenté par avion / discipline manque pour la planification.

UI

Sélection → "Charger" remplit jumpersList en dépliant la composition (un objet PARA_TYPOLOGIES cloné par count × type) puis appelle buildJumpers() + recompute().

Bénéfice optimisation

Avec un stick type chargé, l'optimiseur résout un problème représentatif :

  • Tracking → axe contraint par leur direction d'évacuation.
  • Wingsuit → côté de vol cohérent (wsSide propagé).
  • Tandems → marges RDV plus fines (voile chargée, descente rapide).
  • Élèves → marges minimales et dtSortie plus longs.

L'axe / offset / ordre trouvés sont directement applicables le jour J sans relancer l'optimisation pour le stick effectif (sauf composition très différente).

Bonus — Statistiques de stick types. Pour planification, faire tourner l'optimiseur sur les 6 sticks types en parallèle (un worker chacun) avec une matrice météo (axe vent × vitesse vent), et présenter une carte de chaleur stick × condition → score moyen. Aide à choisir l'avion pour la rotation. Documentaire pour l'instant — hors scope immédiat.

Fichiers. app.js (constante STICK_PRESETS + chargement), builder UI dans buildJumpers, sélecteur HTML près de la zone parachutistes.
